# 编排决策准则：Subagent / Agent Team / Skill / Slash Command 怎么选

> 高频决策：手上这件事，到底该派个分身、开个团队、写个命令、还是封个技能？
> 本文给出**决策树 + 判据表 + 反模式**。原则服从 [`../AGENTS.md`](../AGENTS.md) 第 3 节（计算资源）与第 6 节（决策原则）。

---

## 0. 四者本质（一句话）

| 机制 | 本质 | 一句话 |
|------|------|--------|
| **Slash Command** | 参数化的提示词快捷方式 | 把**简单**的重复流程一键化 |
| **Skill** | 按需披露的专业知识包（`SKILL.md` + 可选脚本/资源） | 把**复杂**的专业流程封装、用时才加载 |
| **Subagent** | 当前会话派出的"分身"，独立上下文，只回摘要 | 派出去干独立重活，**省主上下文** |
| **Agent Team** | 多个**完全独立**的 agent 实例，可互发消息、共享任务列表 | 多人并行 + 互相挑战，**贵** |

记忆锚点：**Command/Skill 管"规则与流程"，Subagent/Team 管"干活"。**

## 1. 主决策树

```
我现在面对的是什么？
│
├─ A. 一个【可复用的流程/知识】，想沉淀下来供以后反复用
│     │
│     ├─ 简单（几步、靠提示词就能描述清楚，无需额外脚本/大量知识）
│     │      → 写一个 Slash Command          （模板：command.template.md）
│     │
│     └─ 复杂（多步、需领域知识/参考资料/可执行脚本、需按需加载）
│            → 封装一个 Skill                 （模板：skill.SKILL.template.md）
│
└─ B. 一件【当下要执行的活】
      │
      ├─ 能拆成若干【互相独立】的子任务，结果导向、不用频繁来回
      │      │
      │      ├─ 子任务之间【无需讨论/互相挑战】，只要各自给结果
      │      │      → 派多个 Subagent 并行（只读探查首选 explore subagent）
      │      │
      │      └─ 子任务需要【互相验证 / 竞争假说 / 跨模块协作】
      │             → 开 Agent Team（3–5 人，慎用，成本高）
      │
      └─ 任务【集中在一两个文件 / 需频繁来回 / 对延迟敏感】
             → 自己直接干，别派分身（启动 subagent 有重建上下文的开销）
```

## 2. Subagent vs Agent Team 判据表

| 维度 | Subagent | Agent Team |
|------|----------|-----------|
| 本质 | 当前会话的分身 | 多个独立实例 |
| 通信 | 只把摘要回报给 lead | 成员之间可直接互发消息 |
| 协调 | Lead 全程管 | 共享任务列表，自行认领 |
| Token 成本 | **低**（只回摘要） | **高**（每人独立实例） |
| 适合 | 任务独立、结果导向 | 需讨论、相互验证、跨模块并行 |
| 规模建议 | 按需多个并行 | **3–5 人**，每人 ≤ 5–6 任务 |

**用 Subagent 的典型场景** ✅
- 跑测试/构建等海量输出任务 → 只回传失败摘要，保护主上下文。
- 多个无依赖的检索/调研并行（如同时查 3 个模块的实现）。
- 流水线：审查 subagent 找问题 → 修复 subagent 改。

**用 Agent Team 的典型场景** ✅
- 并行多视角代码审查（安全 / 性能 / 测试覆盖各一人）。
- 竞争假说调试（多 agent 互相挑战，逼出真正根因）。
- 跨模块并行开发（前端 / 后端 / 测试各一人，**按文件或模块严格切分，禁止并发写同一文件**）。

**不要用 Subagent/Team 的场景** ❌
- 改动只集中在一两个文件。
- 需要频繁来回调整、共享大量上下文。
- 对响应速度敏感（启动开销不划算）。

## 3. Slash Command vs Skill 判据表

| 维度 | Slash Command | Skill |
|------|---------------|-------|
| 复杂度 | 简单、线性、几步内 | 复杂、多步、有分支 |
| 是否需额外知识/资料 | 否，提示词够用 | 是，需打包领域知识/参考 |
| 是否需可执行脚本 | 一般否 | 可附带脚本/工具 |
| 加载方式 | 用户主动 `/调用` | Agent 按描述**自动发现并按需加载** |
| 维护成本 | 低 | 中高 |
| 典型例子 | `/changelog`、`/new-component`、`/lint-fix` | "处理 PDF 表单"、"按公司规范出季度报告"、"操作某内部平台" |

**判断口诀**：
> "我能用一段提示词把它说清楚吗？" → 能 → **Command**；
> "它需要一整套知识/资料/脚本，且我希望 agent 在合适时机自动想起它" → **Skill**。

## 4. 沉淀的触发时机（动态发现 SOP）

不要预先猜哪些要沉淀。**在运行中观察**，命中以下信号才动手：

- 同一类操作**手动做了 ≥ 3 次** → 该沉淀了。
- 每次都要复述一长串相同上下文/规则 → 沉淀。
- 某流程一旦做错代价很大、需要强制对齐步骤 → 沉淀（并尽量机械化校验）。

沉淀后：在目标项目 `LOG.md` 记一笔"为什么沉淀、沉淀成了什么"，并在 `AGENTS.md` 的编排路由里登记，方便后续 agent 发现。

## 5. 反模式（Anti-patterns）

- ❌ **过度并行**：为了"显得高效"无脑开 team，结果 token 爆炸、成员互相覆盖文件。
- ❌ **该串行却并行**：让两个 agent 同时写同一文件 → 互相覆盖。
- ❌ **简单事做成 skill**：一句提示能搞定的非要封装，增加维护负担（熵增）。
- ❌ **沉淀过早**：只做过一次就急着抽象，往往抽错。先观察重复信号。
- ❌ **subagent 干琐碎活**：改一两行还派分身，启动开销 > 收益。
- ❌ **skill 一直膨胀**：定期精简，harness 应趋向简单。
