# SOP：为目标项目从 0 搭建 Harness System

> 适用场景：用户把一个**目标项目仓库**交给你，要你"给它搭一套 harness / agent 工作框架"。
> 本文是确定性流程（Blueprint）。按阶段执行，每阶段都有**退出条件**，不满足不得进入下一阶段。
> 全程受 [`../AGENTS.md`](../AGENTS.md) 红线约束（尤其 R3：只在目标项目目录内写）。

---

## 阶段 0 · 探查（Discover）—— 只读，禁止写

**目标**：在动手前对目标项目形成"全景图"。**此阶段一律只读**，可大量并行 / 派 explore subagent。

要查清并记录：
- [ ] 目标项目根路径、是否 git 仓库、当前分支与状态。
- [ ] 项目用途 / 背景（看 `README`、文档、issue、commit 历史）。
- [ ] **技术栈**：语言、框架、依赖管理文件、运行时版本。
- [ ] **构建 / 测试 / 运行命令**（精确到可复制粘贴，例如 `pnpm test`、`make build`）。
- [ ] 既有规范：linter / formatter 配置、目录约定、命名风格。
- [ ] 是否已有 harness 痕迹（已存在的 `AGENTS.md`/`CLAUDE.md`/`.cursorrules` 等）——若有，**增量改进而非推翻**。
- [ ] 计算资源约束（是否有远端机器、GPU、CI、配额限制）。
- [ ] 潜在红线（涉密数据、生产环境、外网/内网访问、密钥位置）。

**退出条件**：能用三句话讲清"这是什么项目、用什么技术、怎么跑起来"。否则继续探查，不要瞎猜。

---

## 阶段 1 · 规划（Plan）

**目标**：确定要为该项目生成哪些 harness 产物，以及它们的目录落点。

标准产物清单（按需裁剪，模板在 [`../templates/`](../templates/)）：

| 产物 | 作用 | 模板 | 必需? |
|------|------|------|-------|
| `AGENTS.md` | 总指挥：背景/技术栈/计算资源/红线/规范/决策原则 | `AGENTS.template.md` | ✅ 必需 |
| `TODO.md` | 机器可读交接班记忆 | `TODO.template.md` | ✅ 必需 |
| `LOG.md` | 人类可读交接班日志 | `LOG.template.md` | ✅ 必需 |
| `ideas.md` | 灵感库（可做但未做） | `ideas.template.md` | 推荐 |
| `references/` | 参考文档、外部调研、设计资料 | `references.README.md` | 推荐 |
| `skills/` 或 `commands/` | 沉淀的可复用流程 | `skill.SKILL.template.md` / `command.template.md` | 按需 |

**目录落点原则**：
- 优先放在目标项目**根目录**（`AGENTS.md` 是跨工具事实标准，放根目录可见性最高）。
- 大型 monorepo 可在**每个子系统**放一个 `AGENTS.md`（OpenAI 仓库有 88 个），根目录那份做路由。
- 不与目标项目既有同名文件冲突；冲突时先读再增量合并。

**退出条件**：产出一份"要创建哪些文件、各放哪里"的清单（可直接写进目标项目的 `TODO.md` 作为初始待办）。

---

## 阶段 2 · 生成（Build）

按以下顺序生成（有依赖关系，**串行**）：

1. **先 `AGENTS.md`**：从模板复制，**逐项用阶段 0 的探查结论填实**。
   - 背景：用目标项目自己的话。
   - 技术栈 + 命令：**必须精确**，每条命令都应可直接运行。
   - 红线：继承通用红线（见 `AGENTS.template.md`），再叠加该项目特有红线（如"不许碰生产 DB"）。
   - 代码规范：沿用项目既有规范优先。
   - 决策原则 + 编排路由（subagent/team/skill/command）。
2. **再 `TODO.md` / `LOG.md`**：用模板初始化。`TODO.md` 写入阶段 1 的产物清单 + 目标项目首批真实任务；`LOG.md` 写第一条"harness 初始化"记录。
3. **`ideas.md` / `references/`**：把探查中冒出的、暂不做的想法塞进 `ideas.md`；把找到的有用外链/设计资料归档进 `references/`。
4. **skills / commands**：此阶段一般先不建，留到运行中发现重复劳动再沉淀（见 [`orchestration.md`](./orchestration.md)）。

> ⚠️ 所有文件**只写在目标项目目录内**（红线 R3）。密钥只走环境变量（红线 R4）。

---

## 阶段 3 · 闭环自验证（Verify）—— 对照需求，不是对照自己

**目标**：交付前确认这套 harness 真的能让"下一个 agent"接手。做一次"换位测试"：

- [ ] 假装自己是个**对该项目一无所知的新 agent**，只读你生成的 `AGENTS.md` + `TODO.md` + `LOG.md`，能不能知道"现在该干什么、怎么跑起来、什么不能碰"？任一不行 → 回阶段 2 补。
- [ ] `AGENTS.md` 里每条命令是否都**真的能跑通**（能验证就跑一次只读命令验证）？
- [ ] 红线是否覆盖了阶段 0 发现的真实风险？
- [ ] 文档是否过长 / 重复 / 含空泛废话（熵治理：精简掉）？
- [ ] 是否误碰了 harness 之外的文件（红线 R3 复核）？

**退出条件**：换位测试通过。否则不得宣布完成。

---

## 阶段 4 · 记账 + 交接（Record）

1. 在**目标项目**的 `TODO.md` 勾掉"搭建 harness"任务，并列出建议的后续首批任务。
2. 在**目标项目**的 `LOG.md` 追加一条：做了什么、为什么这么设计、留下哪些待决问题。
3. 在**本仓库**的 [`../LOG.md`](../LOG.md) 追加一条：为哪个项目搭了 harness、用了哪些模板、踩了什么坑、有什么可反哺到模板/本 SOP 的改进。
4. 若过程中发现本 SOP / 模板有缺陷 → **顺手改进它们**（这是 harness-over-harness 的自我进化）。

---

## 阶段 5 · 持续演进（Evolve，运行期）

目标项目跑起来后，harness 不是一锤子买卖：
- Agent 每犯一次错 → 往目标项目 `AGENTS.md` 加一条规则（文档免疫系统）。
- 发现反复手动做的流程 → 沉淀为 command（简单）或 skill（复杂），详见 [`orchestration.md`](./orchestration.md)。
- 定期回看：删掉过时/腐烂的规则，保持 harness 简单（熵治理）。

---

## 速查：最小可交付（MVP Harness）

时间紧时，**至少**交付这三件，缺一不可：
1. `AGENTS.md`（含真实可跑的命令 + 红线）
2. `TODO.md`
3. `LOG.md`
